# 产品需求(PRD)— 迭代:页面美化 / 体验升级

> 起草于**需求澄清官**(issue CUI-14,2026-05-30),由**产品经理**细化为可交付 PRD + 产品级 UI 方向。
> 标注:`[已确认]` = 客户已拍板;`[产品定稿]` = 产品经理在客户授权范围内给出的可执行决策(可在 02-design 草样阶段微调);`[需架构对齐]` = 产品定口径,实现/性能细节由架构师在 02-design 拍板。
> 状态 `done`:产品需求与 UI 方向定稿;**主题草样**的客户"过样"在 **02-design** 阶段进行(架构/设计出可视稿后给客户过目)。转架构师做 UI/响应式/动画/音效方案。

## 1. 背景与问题
首版已交付可玩,但前端**观感与体验薄弱**(客户截图):暗色极简、**棋盘超出一屏需滚动**、棋盘中央空、**看不到棋子/不知道自己在哪**、**无掷骰动画**、地产格视觉单薄、无音效。本迭代解决"好看 + 好用 + 有大富翁氛围",**只动前端表现层**。
> 现状基线(供参考):单页 `public/`(index.html / app.js / style.css),分 entry / lobby / game / result 四区;game 为两栏 `.game-grid`(棋盘 8×8 网格 + 右侧 players/actions/subpanel/log),`@media(max-width:820px)` 退化为单列;棋子为 10px 彩色圆点,当前格金色描边;暗色 GitHub 风。

## 2. 目标 / 非目标
- **目标 [已确认]**:对 Web 前端做视觉与交互升级,达成 §3 的 6 个需求点,且**桌面 + 手机竖屏均一屏无滚动**。
- **非目标 [已确认]**:
  - **不改玩法规则、对局逻辑、数值**(沿用首版);
  - **不改后端协议与服务端权威裁决**(掷骰仍服务端,前端只做动画表现);
  - 不做微信小程序端(仍 Web/H5;但响应式须覆盖手机浏览器);
  - 不做商业化/匹配/账号等(见 baseline PRD 非目标)。

## 3. 核心需求(按优先级)
| 优先级 | 需求 | 说明 |
|--------|------|------|
| P0 | **一屏自适应(含手机)** [已确认] | 桌面浏览器**与手机竖屏**都能**一屏完整呈现**棋盘 + 当前操作区,**无需滚动**(客户最大痛点) |
| P0 | **直观显示位置** [已确认] | 棋盘上可见各玩家棋子,**自己的棋子高亮**;移动有**沿格移动动画** |
| P0 | **掷骰动画** [已确认] | 掷骰有动画;**最终点数以服务端事件为准**(前端只演出,不自行决定结果) |
| P0 | **整体视觉美化 + 丰富大富翁元素** [已确认] | 风格 = **B 丰富卡通 / 轻拟物自创主题**;地产色带、建筑(房子/旅馆)、卡牌、角色棋子 |
| P0 | **音效** [已确认] | 关键动作音效;**可静音 / 音量控制** |
| P1 | **去 IP 化(硬约束)** [已确认] | 自创主题皮肤,不照搬 Monopoly 受保护元素;命名/视觉自创 |
| P1 | **结果页 / 大厅 / 进房 同步美化** [产品定稿] | §3F:全流程视觉一致,避免"对局好看、其余仍旧" |

## 3A. 主题方向(§5①答复)[产品定稿,02-design 出草样请客户过样]
> 客户已授权团队自行选定主题。产品**推荐一个主方向 + 2 个备选**,由 02-design 出可视草样,客户过样后定稿。

- **主推:🏙️「环游都市」现代都市轻拟物卡通**
  - 理由:最贴合大富翁"买地/盖楼/收租"心智(地产=街区地标/店铺,建筑=小房子→公寓→地标塔,自然对应房子/旅馆升级);受众最广、易去 IP(用自创城市/街区名,避开经典棕橙配色与 HOTEL 字样);色彩可做明快卡通。
- **备选 A:🏝️ 海岛度假**(轻松明快、配色讨喜);**备选 B:🚀 太空殖民**(差异化、年轻向)。
- **去 IP 复核口径**:配色不照搬经典大富翁色序;不用 "Monopoly / 大富翁 / HOTEL" 等受保护字样与标志性图形;卡牌文案、货币名、地名全自创。

## 3B. 响应式布局规格(§5②答复)[产品定稿 / 部分 ★需架构对齐]
**核心铁律:棋盘 + 当前回合操作 一屏内无滚动可完成一回合。** 次要信息(完整事件日志、玩家明细、交易/拍卖详情)允许收纳进抽屉/弹层,不得把主操作挤出屏幕。

- **断点**:桌面 ≥ 820px 维持现有两栏;< 820px 单列;**手机竖屏 ≤ 480px 启用"底部操作"形态**。
- **棋盘**:保持正方形(`aspect-ratio:1/1`),尺寸按可视区取 `min(可用宽, 可用高)` 自适应缩放,**优先缩放、不重排**(8×8 结构不变,保证位置直觉一致);棋盘居中,**中央空区**放主题插画 + 当前回合/计时/掷骰按钮(顺带解决"中央空"问题)。
- **手机竖屏区域划分(自上而下)**:① 顶部精简状态条(房间码、连接、静音、当前回合玩家+倒计时);② 棋盘(占主视区);③ 底部**固定操作条**(掷骰/买地/结束回合等当前可用动作)。
- **次要信息收纳**:
  - **玩家列表** → 竖屏顶部横向头像条(点开看明细);
  - **交易 / 拍卖 / 卡牌等 subpanel** → **底部弹出面板(bottom sheet)/ 全屏弹层**,操作完成即收起(它们是瞬时强交互,适合弹层,不常驻);
  - **事件日志** → 竖屏默认**折叠**,可点开/上滑查看(次要信息)。
- ★ 具体断点像素、bottom-sheet 交互与手势、横屏处理由架构师在 02-design 定。

## 3C. 动画程度与降级(§5③答复)[产品定稿 / ★需架构对齐性能策略]
- **三档动画**:
  - **完整(默认,桌面/中高端机)**:棋子沿格逐格缓动移动、掷骰滚动/翻转动画、买地/收租浮字与金币动效、卡牌翻转;
  - **精简(低端机 / 弱网 / 系统省电)**:棋子直接渐隐位移到目标格、掷骰用快速数字滚动后定格、去粒子/复杂特效;
  - **关闭(手动或 `prefers-reduced-motion`)**:状态瞬时切换,仅保留必要反馈。
- **降级触发**:**必须尊重系统 `prefers-reduced-motion`**;提供设置项手动切换;按设备/帧率粗判自动降级的策略 ★ 交架构。
- **掷骰动画硬口径(防作弊/一致性)**:最终点数 **= 服务端 `DiceRolled` 事件值**;动画仅为演出,**收到服务端结果后必须定格到该点数**,动画不得"先随机播完再硬改",也不得因动画未播完而延迟呈现服务端结果(★ 动画与事件对齐方式交架构)。

## 3D. 音效(§5④答复)[产品定稿]
- **清单(关键动作)**:掷骰(滚动+落定)、棋子移动(轻 tick)、买地/成交、收租(金币)、抽卡、入狱、破产(失败音)、结算(胜利音)、轮到你(提示音)、按钮轻点。统一**短促卡通风**。
- **素材来源**:**免版权(CC0)** 音效库(如 freesound CC0 / kenney.nl 免费音效)或自制;**禁用**有版权/带 IP 的音效。素材随仓库放 `public/`,体积可控(优先压缩后的 mp3/ogg)。
- **开关与记忆**:header/状态条放**静音 + 音量**控制;默认开(可较低音量);用 `localStorage` 记忆用户选择。
- ★ 浏览器/移动端音频策略:首次播放需用户手势解锁(autoplay policy),架构师在 02-design 处理首次交互解锁与多端兼容。

## 3E. 棋子 / 玩家配色与头像(§5⑤答复)[产品定稿]
- **8 色高对比调色板**:为 2–8 人提供 8 个**易区分**的颜色,**避免仅靠红绿区分**(色盲友好);**颜色 + 形状/角色图标双重区分**,不单靠颜色。
- **自己的棋子高亮**:除颜色外加**描边/光晕 + "你"标记**,并在玩家条标注自己,确保"一眼知道我在哪"。
- **头像/棋子造型**:与主题一致的**自创卡通角色**(如都市主题:小人/小车/帽子等),首版用一组**预设角色图标**(无需用户上传);多人同格时小型化叠放,自己的置顶。

## 3F. 美化范围(§5⑥答复)[产品定稿]
**纳入本迭代,保证全流程视觉一致**:对局页(棋盘/侧栏/弹层)、**结果页**(排名/净资产/胜者高亮 + 再来一局)、**大厅 lobby**(房间码/链接/座位)、**进房 entry**、连接状态条——统一按选定主题美化。

## 4. 成功指标 / 验收方向
- **一屏**:1366×768、1920×1080 **与** 手机竖屏(如 390×844)下,棋盘 + 当前操作区**一屏完整、无需滚动**即可完成一回合。
- **位置可见**:任意时刻一眼看出自己与各玩家所在格;掷骰后棋子有可见**移动动画**到目标格;自己棋子明显高亮。
- **掷骰动画**:点击有动画,落定点数 = 服务端 `DiceRolled` 值(不偏离服务端权威)。
- **视觉**:成套**自创主题**(丰富卡通/轻拟物),地产/建筑/卡牌/棋子有辨识度;**去 IP 复核通过**(无 Monopoly 商标元素)。
- **音效**:关键动作有音效且**可静音**,刷新后记住开关。
- **降级**:`prefers-reduced-motion` 下动画自动收敛,弱设备仍可玩。
- **不回归**:玩法/对局逻辑与首版一致,后端协议未改,**`npm test` 仍 38 全绿**(只动前端,不破坏既有测试)。

## 5. 待确认(细化后剩余)
> 原 6 项已在 §3A–§3F 给出产品定稿;此处仅留需客户过样或需架构深化的:
- [ ] **[客户过样 · 02-design]** §3A 主题方向:推荐「环游都市」,**草样在 02-design 出可视稿后请客户过目/选定**(客户已授权团队先行,非阻塞)。
- [ ] **[需架构对齐 · 02-design]** §3B 断点与 bottom-sheet 交互、§3C 动画与事件对齐 + 性能降级策略、§3D 音频解锁与多端兼容。

## 6. 约束(硬,沿用 baseline)
- **纯前端(`public/`)改动**;**不碰 `src/engine`(规则)与 `src/gateway`(协议/裁决)**——端无关 + 服务端权威沿用 baseline `02-design`。
- **去 IP 化**(P1 硬约束)。
- **掷骰点数以服务端为准**(前端只演出)。
- **不回归既有 38 测试**。
- 平台:Web/H5,响应式覆盖桌面 + 手机浏览器;小程序端不在本迭代。

## 7. 关键产品决策小结(给架构师 02-design 起步)
1. **一屏优先级**:棋盘 + 当前操作 = 必须常驻;日志/玩家明细/交易拍卖详情 = 可收纳。这条决定整套响应式取舍。
2. **棋盘缩放不重排**:保留 8×8 结构,靠等比缩放适配,保证位置直觉与多端一致。
3. **中央空区利用**:放主题插画 + 当前回合/掷骰区,一举解决"中央空"和"掷骰入口显眼"。
4. **动画三档 + 尊重系统偏好**,掷骰动画严格对齐服务端事件值。
5. **主题先出草样再定**:产品给方向,02-design 出可视稿走客户过样。
