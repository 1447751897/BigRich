# 产品需求 — 迭代:页面美化 / 体验升级

> 由**需求澄清官**基于与客户确认的澄清(issue CUI-14,2026-05-30)填充。
> `[已确认]` = 客户已拍板;`[假设待确认]` = 澄清官合理补全,待产品/设计核实。
> 状态 `in_review`:待产品经理细化为 PRD、产出 UI 方案后推进。

## 1. 背景与问题
首版已交付可玩,但前端**观感与体验薄弱**(见客户截图):暗色极简、**棋盘超出一屏需滚动**、棋盘中央空、**看不到棋子/不知道自己在哪**、**无掷骰动画**、地产格视觉单薄、无音效。本迭代专门解决"好看 + 好用 + 有大富翁氛围"。

## 2. 目标 / 非目标
- **目标 [已确认]**:对 Web 前端做视觉与交互升级,达成下列 6 个需求点。
- **非目标 [已确认]**:
  - **不改玩法规则、对局逻辑、数值**(沿用首版);
  - **不改后端协议与服务端权威裁决**(掷骰仍服务端,前端只做动画表现);
  - 不做微信小程序端(仍 Web/H5;但要求响应式覆盖手机浏览器);
  - 不做商业化/匹配/账号等(见 baseline PRD 非目标)。

## 3. 核心需求(按优先级)
| 优先级 | 需求 | 说明 |
|--------|------|------|
| P0 | **一屏自适应(含手机)** [已确认] | 桌面浏览器 **与手机竖屏**都能**一屏完整呈现**棋盘+操作区,**无需滚动**(客户明确点出的最大痛点)。 |
| P0 | **直观显示位置** [已确认] | 棋盘上可见各玩家**棋子**,**自己的棋子高亮**;移动时有**沿格移动动画**。 |
| P0 | **掷骰动画** [已确认] | 掷骰有动画表现;**最终点数以服务端事件为准**(前端只演出,不自行决定结果)。 |
| P0 | **整体视觉美化 + 丰富大富翁元素** [已确认] | 风格方向 = **B:丰富卡通 / 轻拟物的"大富翁感"自创主题**;地产分组色带、建筑(房子/旅馆)、卡牌、角色棋子等视觉元素。客户授权"**可自行参考网上元素进行页面风格重构**"。 |
| P0 | **音效** [已确认] | 掷骰、成交/买地、收租、破产、结算等关键动作音效;**需可静音/音量控制**。 |
| P1 | **去 IP 化(硬约束,沿用首版)** [已确认] | "参考网上元素"= 自创主题皮肤,**不得照搬 Monopoly 受保护元素**(棋盘配色/HOTEL/具体卡面文案等),命名/视觉自创。 |

## 4. 成功指标 / 验收方向
- **一屏**:在主流桌面分辨率(如 1366×768、1920×1080)**与**主流手机竖屏(如 390×844)下,棋盘 + 当前操作区**一屏内完整可见、无需滚动**即可完成一回合操作。
- **位置可见**:任意时刻能一眼看出**自己**与各玩家所在格;掷骰后棋子有可见的**移动动画**到目标格。
- **掷骰动画**:点击掷骰有动画,落定点数 = 服务端 `DiceRolled` 事件值(不偏离服务端权威)。
- **视觉**:呈现成套**自创主题**(丰富卡通/轻拟物),地产/建筑/卡牌/棋子有辨识度;**无 Monopoly 商标元素**(去 IP 化复核通过)。
- **音效**:关键动作有音效且**可静音**。
- **不回归**:玩法/对局逻辑与首版一致,后端协议未改,`npm test` 仍 **38 全绿**(本迭代只动前端,不应破坏既有测试)。

## 5. 待确认(交产品/设计细化,不阻塞起步)
- [ ] **具体主题选定**(城市 / 海岛 / 太空 / 国潮 / 美食…):客户授权团队自行选,产品/设计定一个方向并可出草样给客户过目。
- [ ] **手机竖屏布局形态**:棋盘缩放 vs 重排;操作区(交易/拍卖面板)在小屏如何收纳(抽屉/弹层/Tab)。
- [ ] **动画程度与性能**:移动/掷骰/成交动画的力度,低端手机的降级策略。
- [ ] **音效清单与素材来源**(免版权/自制)、默认开关与音量记忆。
- [ ] **棋子/玩家配色与头像**风格。
- [ ] 是否需要"再来一局/结果页"等界面的同步美化(建议纳入,保持一致)。

## 6. 约束
- **纯前端(`public/`)改动**;不碰 `src/engine` 与 `src/gateway` 协议/裁决(端无关、服务端权威沿用 baseline `02-design`)。
- **去 IP 化**(P1 硬约束)。
- 平台:Web/H5,**响应式覆盖桌面 + 手机浏览器**;小程序端不在本迭代。
