# 方案设计 — 迭代:页面美化

> 输入:本迭代 `01-requirements.md`(PRD,done)+ baseline `project/02-design.md`(端无关+服务端权威)。
> 本文回应 PRD §5 两项遗留:**[客户过样]主题草样**(§2 出可视稿)与 **[需架构对齐]** 5 项(响应式断点/手势、动画三档与事件对齐、音频解锁、增量选型、主题草样)。
> 硬约束(继承 PRD §6):**只动 `public/` 渲染层**,不碰 `src/engine`、`src/gateway`;**去 IP 化**;**掷骰点数 = 服务端 `DiceRolled` 事件值**;**不回归既有 38 测试**。

## 1. 总体思路
**纯渲染层重做,零协议改动、零依赖膨胀。** 客户端仍是"瘦客户端":只消费服务端 `state`+`events`,所有规则结果以服务端为准(掷骰尤其)。本迭代把现有 `public/{index.html,app.js,style.css}` 从"暗色极简能玩"升级为"自创卡通主题、双端一屏、有位置感/动画/音效"。

关键取舍:
- **缩放不重排(P0 铁律)**:棋盘永远 8×8 同结构,靠 `min(可用宽,可用高)` 等比缩放适配,**绝不在小屏重排格子顺序**——保证多端位置直觉一致(PRD §3B/§7.2)。
- **CSS 优先、canvas 最小化、零新依赖**:动画与布局用 CSS(transform/transition)+ 少量 JS 编排;掷骰骰子用 CSS 3D 即可,**不引入 howler/动画库**(详见本迭代 `03-tech-stack`,D-101)。音频用原生 Web Audio API。
- **动画是"演出层",权威态是"事实层",两者解耦**:动画播放前先吃下服务端已下发的最终结果,动画只负责把状态从 A 演到 B,**任何时刻被新 state 打断都能立刻定格到权威值**(详见 §4.3 掷骰对齐)。
- **渐进增强 + 可降级**:三档动画(完整/精简/关闭)+ 尊重 `prefers-reduced-motion`;弱设备/弱网自动降级仍可玩。

## 2. 视觉/主题方案(草样,待客户过样)

> 我无法在 markdown 内直接出位图;下面给**主推 + 2 备选**的**可视规格 + ASCII 线框草样**作为过样依据,客户选定方向后,实际皮肤在 04-mvp 落地(可先出一张静态 mock 截图补充)。**推荐:🏙️ 环游都市。**

### 2A. 主推:🏙️「环游都市」现代都市轻拟物卡通
- **隐喻映射**:地产=街区地标/店铺;房子 1→4=小屋→联排→公寓→写字楼;旅馆=地标塔;角=市政广场(起点)/拘留所(监狱)/中央公园(免费停车)/警局(入狱);机会=「机遇」信封卡、命运=「事件」喇叭卡、税=「市政」收据。**全部自创命名,规避 Monopoly/HOTEL/大富翁 字样与经典棕橙色序**(去 IP 复核口径见 PRD §3A)。
- **色彩**:明快卡通、轻拟物(柔和阴影+圆角+微高光),不拟真。中央空区放都市天际线插画 + 掷骰区。
- **6 个地产分组色带**(沿用引擎现有 groupId,仅改 hex 为卡通明快版,**不改引擎**):青 teal / 琥珀 amber / 珊瑚 coral / 紫罗兰 violet / 靛蓝 indigo / 绯红 crimson。

### 2B. 备选
- **🏝️ 海岛度假**:沙滩/码头/度假村,蓝绿暖沙配色,轻松讨喜。
- **🚀 太空殖民**:星球/空间站/殖民舱,深空+霓虹,年轻向差异化。
> 三者共用同一套布局/交互/动画/音效框架,**仅皮肤(配色/插画/命名/图标)不同**——主题切换成本低,过样后改一套主题变量即可。

### 2C. 棋子/玩家区分(PRD §3E,色盲友好)
- **8 色高对比调色板** + **形状/角色图标双重编码**(不单靠颜色,避免仅红绿区分)。建议 8 个角色图标:🚕出租车 / 🎩礼帽 / 🐶小狗 / 🚀火箭 / ⛵帆船 / 🌂雨伞 / ⚽足球 / 🎸吉他(随主题换造型,形状各异)。
- **"自己"强标识**:自己的棋子加**描边光晕 + "你"角标**,玩家条里自己置顶/加亮;多人同格小型化叠放、自己置顶。

## 3. 响应式 / 一屏布局(★ PRD §3B)

**铁律:棋盘 + 当前回合操作 一屏内无滚动可完成一回合;次要信息(完整日志/玩家明细/交易拍卖详情)收纳进抽屉/弹层。**

### 3A. 断点(拍板)
| 断点 | 形态 | 布局 |
|------|------|------|
| **≥ 900px 桌面** | 两栏 | 左:棋盘(等比方形,取 `min(可用宽-侧栏, 可用高)`);右:固定侧栏(玩家条 + 操作区 + 可折叠日志)。(现有 820px 上调到 900,给侧栏更稳空间) |
| **600–899px 平板/窄窗** | 单列 | 棋盘居中等比;操作区移到棋盘下方紧贴;玩家条横向;日志折叠 |
| **≤ 599px 手机竖屏** | 三段式 | ① 顶部精简状态条;② 棋盘占主视区(`min(100vw, 可用高)`);③ **底部固定操作条**;次要交互走 **bottom-sheet** |

> 用 `dvh`(动态视口高)而非 `vh`,规避移动端地址栏伸缩导致的"差一点要滚动"。棋盘容器 `aspect-ratio:1/1` + `max-block-size`/`max-inline-size` 双约束取小边。

### 3B. 手机竖屏区域(自上而下)
```
┌──────────────────────────────┐
│ 房间码 RM3K · 🟢 · 🔊 · ⏱22s  │  ① 顶部状态条(连接/静音/倒计时)
│ 🚕你 🎩A 🐶B 🚀C  (横向头像条)│     玩家头像条(点开看明细→bottom-sheet)
├──────────────────────────────┤
│        [ 8×8 棋盘等比 ]       │  ② 主视区:棋盘缩放不重排
│      中央:天际线 + 🎲掷骰     │     中央空区放主题插画 + 掷骰按钮/动画
│                              │
├──────────────────────────────┤
│  [🎲 掷骰]  [结束回合]  [⋯]   │  ③ 底部固定操作条(当前可用动作)
└──────────────────────────────┘
     交易/拍卖/盖房 → 从底部弹出 bottom-sheet,操作完收起
     完整日志 → 顶部"事件"入口或上滑,默认折叠
```

### 3C. bottom-sheet 交互/手势(拍板)
- **触发**:点"我的地产/交易"或服务端把我推入 `turnPhase=trade(awaiting=我)`/`auction(轮到我)` 时**自动弹出**对应 sheet(强交互、限时,必须立刻可见)。
- **手势**:向下拖拽关闭(非限时类如"我的地产管理");**限时类(交易响应/拍卖出价)不允许误触关闭**——只能通过"接受/拒绝/出价/放弃"按钮收起,避免手势误关导致超时被动判负。
- **遮罩**:sheet 弹出时半透明遮罩盖住棋盘但**保留顶部倒计时可见**(让玩家知道还剩多少秒)。
- **横屏**:手机横屏视作"窄桌面",走单列/两栏的中间形态;不做专门横屏棋盘(等比缩放已覆盖);若高度过低(<420px)提示"竖屏体验更佳"但不强制。

### 3D. 桌面/平板
桌面维持两栏但棋盘 tile 内元素(地产名/价/色带/房子/棋子)按 tile 实际像素做 `clamp()` 字号,避免缩放后文字溢出(现有 `font-size:10px` 固定值在大屏偏小、小屏溢出)。

## 4. 关键交互 / 动画(★ PRD §3C)

### 4A. 三档动画 + 实现路径
| 档位 | 触发 | 棋子移动 | 掷骰 | 特效 | 实现 |
|------|------|---------|------|------|------|
| **完整**(默认,中高端) | 默认 | 沿圈逐格缓动(每格 ~120ms) | CSS 3D 骰子滚动→定格 | 买地/收租浮字、金币、卡牌翻转 | CSS `transform/transition` + JS 编排队列 |
| **精简**(低端/弱网/省电) | 自动或手动 | 渐隐直接位移到目标格(单段 ~200ms) | 数字快速滚动→定格 | 去粒子,仅保留浮字 | 同上,关掉中间帧 |
| **关闭**(`prefers-reduced-motion`/手动) | 系统偏好或手动 | 瞬时切换位置 | 直接显示点数 | 仅必要状态反馈 | 无过渡 |

- **降级判定**:① 启动读 `matchMedia('(prefers-reduced-motion: reduce)')` → 命中即"关闭"档;② 设置项手动三选一(`localStorage` 记忆);③ **自动降级**:运行时用 `requestAnimationFrame` 采样,若连续 N 帧(如 1s 内)平均 FPS < 阈值(如 40),自动从完整降到精简(只降不升,避免抖动),并提示一次。设备粗判(`navigator.hardwareConcurrency`/`deviceMemory` 低)可作初始默认精简。
- **动画编排**:一条 `state` 可能带多个 `events`(掷骰→移动→落点→付租…)。前端维护一个**事件动画队列**:按事件顺序串行播放,**全程不改变最终权威态**(权威态来自 `state`,动画只是从旧渲染过渡到它)。

### 4B. 位置可见 + 移动动画
- 棋子绝对定位到 tile 中心,移动时沿 `ringCoords()` 路径**逐格 translate**(完整档),经过起点有 +奖励浮字(`PlayerMoved.passedStart`)。
- 自己棋子:光晕描边 + "你"角标常驻;当前回合玩家 tile 高亮(沿用 `.cur`,改为更醒目的主题色脉冲)。

### 4C. 掷骰动画与服务端事件对齐(★ 最关键,防作弊/一致性)
**事实来源**:服务端在 `state.events` 里下发 `DiceRolled{ seatId, dice:[a,b], doubles }`,且同批 `state` 已含移动后位置。客户端**永不自行 random 骰子点数**。

对齐算法(收到含 `DiceRolled` 的 state 时):
1. 立即记下权威结果 `dice=[a,b]`(此刻就是"事实",绝不更改)。
2. **完整档**:播放骰子滚动动画(滚动中显示的是随机视觉帧,纯演出),**预设时长上限**(如 700ms);动画结束**强制定格到 `[a,b]`**。
3. **若动画还没播完又来了新 state**(理论上回合制下罕见,但兜底):**立刻中止动画、直接定格当前权威 `[a,b]` 并应用新状态**——绝不让"动画没播完"延迟或改写服务端结果(PRD §3C 硬口径)。
4. 骰子定格后,再启动棋子移动动画(4B);移动结束渲染落点结果(买/租/卡)。
5. **精简/关闭档**:跳过滚动,直接显示 `[a,b]` 与落点。

> 一句话:**动画的最终态恒等于服务端事件值,动画时长不阻塞、不改写权威结果。** 这条是验收硬指标(05-acceptance)。

### 4D. 其它反馈
成交/买地金币动效、收租浮字、入狱/破产/结算的音画反馈(配合 §5 音效);结算页胜者高亮 + 礼花(完整档)/静态奖杯(精简档)。

## 5. 音效方案(★ PRD §3D)

### 5A. 清单与触发(绑定现有事件)
| 音效 | 触发事件 |
|------|---------|
| 掷骰滚动+落定 | `DiceRolled`(滚动起+定格落) |
| 棋子移动 tick | 逐格移动每步(完整档,轻音、限频) |
| 买地/成交 | `PropertyBought` / `TradeResolved(accepted)` / `AuctionResolved(sold)` |
| 收租金币 | `RentPaid` |
| 抽卡 | `CardDrawn` |
| 入狱 | `SentToJail` |
| 破产 | `PlayerBankrupt` |
| 结算 | `GameEnded`(胜利/失败音按自己名次) |
| 轮到你 | `TurnChanged` 且 `seatId===me`(提示音) |
| 按钮轻点 | UI 点击 |

### 5B. 首次播放解锁(★ autoplay policy)
- 浏览器/移动端禁止无用户手势的自动播放。方案:**首个用户手势(点"创建/加入房间"或任意按钮)时初始化并 `resume()` 一个共享 `AudioContext`**,完成解锁;解锁前的音效请求静默丢弃(不报错、不排队堆积)。
- 解锁状态记一个标记;iOS Safari 需在手势同步栈内 `resume`,故绑定在按钮 click 回调里直接做。

### 5C. 开关/音量/素材
- 顶部状态条放**静音切换 + 音量**;默认开、音量适中;`localStorage` 记忆(键如 `bigrich.audio`)。
- **素材**:CC0/自制短促卡通音(freesound CC0 / kenney.nl);放 `public/audio/`,优先压缩 mp3/ogg(各 < ~20KB,总包控制在数百 KB);**禁用任何带版权/IP 的音效**。
- 解码缓存:小文件用 `AudioBuffer` 预解码缓存,移动端弱网首屏不阻塞(懒加载、播放即取)。

## 6. 备选方案与取舍
| 议题 | 选用 | 淘汰 | 理由 |
|------|------|------|------|
| 渲染/动画技术 | CSS transform/transition + 少量 JS 编排 | 整盘 canvas/WebGL 重写 | 现有是 DOM 棋盘,CSS 动画够用、可访问性好、零依赖、改动面小、不碰协议 |
| 动画/音频库 | 原生(CSS + Web Audio) | howler.js / GSAP / anime.js | 需求规模小、`public/` 无构建链,引库增加体积与维护;原生足够(详见 03-tech-stack D-101) |
| 棋盘适配 | 等比缩放不重排 | 小屏重排为竖列/蛇形 | PRD §7.2 铁律:保 8×8 位置直觉与多端一致 |
| 掷骰演出 | 先吃权威值、动画只定格 | 前端先随机播完再改 / 等动画播完再显示 | PRD §3C 硬口径:防作弊 + 不延迟服务端结果 |
| 主题落地 | 一套 CSS 变量 + 资源目录切换皮肤 | 每主题独立页面/分支 | 主推+备选共用框架,过样切换成本低 |

## 7. 风险与未决
- **[需客户过样,非阻塞]** §2 主题方向(推荐环游都市)与备选——草样规格已给,**实际皮肤可视稿在 04-mvp 出**;客户未回也按环游都市默认推进(PRD §5)。
- **[风险] 低端机 8 人同屏 + 逐格动画性能**:已设自动降级(§4A);8 人满载动画压测建议在 05-acceptance 补一条。
- **[风险] 移动端音频解锁兼容(尤其 iOS)**:已定手势内 `resume` 方案(§5B),需在真机回归。
- **[风险] 一屏达标的真机验证**:1366×768 / 1920×1080 / 390×844 三档需在 05-acceptance 实测无滚动(`dvh` 方案需真机确认地址栏伸缩)。
- **[约束自检] 不回归 38 测试**:本迭代只改 `public/`,不动 `src/engine`、`src/gateway`、协议;`npm test` 应保持 38 全绿(验收项)。
