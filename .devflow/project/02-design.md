# 方案设计

> 输入:`project/01-requirements.md`(PRD,status=done)。本文件由**架构师**基于 PRD 落地方案设计。
> 标注约定:`[硬约束]` = PRD 已拍板、设计必须满足;`[设计决策]` = 架构师在本文给出的方案/取舍(关联 03-tech-stack 的 ADR);`[需确认]` = 需产品/客户回确的取舍点。
> 本版重点回应 PRD §4B「交易/拍卖」(回合制里的"实时孤岛")的并发原子性、状态同步与防作弊,以及服务端权威、端无关、三口子预留、断线重连/超时兜底/AI 托管。

## 1. 总体思路

**一句话:服务端权威的"瘦客户端 + 单房间单写者"架构。** 所有规则裁决(掷骰、资金、地产结算、交易/拍卖成交)只在服务端发生;客户端只发"意图(Intent)"、只渲染服务端广播的"事件/状态(Event/State)",不持有任何可影响胜负的逻辑。

三个支撑全局的关键取舍:
1. **单房间单写者(per-room single-writer)[设计决策 / ADR D-001]**:每个房间的全部命令进入**一条有序队列**,由该房间唯一的引擎实例串行处理。这把"交易同价、拍卖超额、并发结算"等竞态从根上消除——命令被串行化,第二个出价一定看到第一个已生效的状态后再校验。熟人量级低并发,无需分布式锁,复杂度最低且天然正确。代价:房间与实例需粘性路由(sticky routing),水平扩展靠"按房间分片",非共享内存并发。
2. **回合主时钟 + 可挂起的子交互时钟,二者互斥不并行 [设计决策 / ADR D-004]**:正常回合用 25s 主时钟;一旦进入交易/拍卖子流程,**主时钟挂起**,子流程跑自己的 20s/10s 时钟;子流程结束再恢复主时钟。任一时刻只有一个时钟在跑,杜绝"双重超时打架"。
3. **核心规则引擎与渲染/传输彻底解耦(端无关)[硬约束 §7.4]**:引擎是纯逻辑——吃 Command、吐 Event,不依赖 WebSocket、不依赖任何端的渲染。Web/H5 与微信小程序共享同一引擎 + 同一套消息协议,各端只写自己的渲染与传输适配层。

## 2. 架构/模块划分

分三层:**客户端(渲染层) — 连接/网关层 — 服务端权威核心**。

```
[Web/H5 客户端]  [微信小程序客户端(后期)]
        \              /
   (统一消息协议:Intent ↑ / Event ↓,JSON,传输无关)
            |
   ┌─────────────────────────────────┐
   │  连接网关层 (Gateway)            │  ← WebSocket;心跳/重连/会话恢复;
   │  - 会话(session=room+seat+token) │     合规拦截 seam(③,首版 no-op)
   │  - 身份抽象(②,首版=匿名昵称)   │
   └───────────────┬─────────────────┘
                   │ (每房间一条有序命令队列)
   ┌───────────────┴─────────────────┐
   │  房间运行时 (Room Runtime)       │  ← 单写者:一个房间=一个引擎实例
   │  ├ 房间/会话管理 (生命周期/观战) │
   │  ├ 回合状态机 (Turn FSM)         │  ← 含子状态:Trade/Auction/Mortgage
   │  ├ 规则引擎 (纯逻辑,端无关)     │     掷骰(服务端RNG)/移动/买地/盖房/
   │  │                               │     租金/机会命运/监狱/破产/结算排名
   │  ├ 定时器服务 (Timer)            │  ← 主时钟+子流程时钟,超时→合成默认命令
   │  ├ AI 托管 (启发式规则)          │
   │  └ 配置/规则集 (§4A 参数化)      │  ← 数值基线作为可配置 RuleSet,不写死
   └───────────────┬─────────────────┘
                   │ (快照)
   ┌───────────────┴─────────────────┐
   │  持久化 (内存主存 + 快照)        │  ← 断线重连/崩溃恢复;低并发够用
   └─────────────────────────────────┘
```

模块职责要点:
- **连接网关层**:管理连接与会话,心跳检测掉线,承载重连与会话恢复;是**身份抽象(②)** 与**合规拦截层(③)** 两个预留口子的落点(见 §5)。它**不含游戏逻辑**。
- **房间运行时**:一个房间一个实例,持有该房间的全部权威状态;所有改动经由单一命令队列串行落地。
- **规则引擎**:纯函数式核心——`(当前状态, 命令) → (新状态, 事件列表)`,不 import 任何传输/渲染/时钟模块。这是端无关与可测试性的关键。掷骰 RNG 只在这里。
- **回合状态机**:管理"轮到谁、当前处于普通回合还是交易/拍卖/抵押子状态";子交互是回合的**嵌套子状态**(§3)。
- **定时器服务**:唯一权威时钟。超时不直接改状态,而是**向命令队列投递一条合成的"默认操作"命令**,与真实玩家命令同路处理——保证引擎是唯一事实源。
- **AI 托管**:按 PRD §4C 的启发式规则生成命令投入队列;托管与真人走同一命令通道。
- **配置/规则集**:§4A 全部数值做成 `GameConfig` 对象,建房时载入,可按房覆盖,便于试玩调参,**绝不写死**。

## 3. 关键流程

### 3.0 命令/事件主循环(贯穿所有流程)
客户端发 Intent → 网关校验会话/权限 → 入房间命令队列 → 引擎串行处理(校验合法性 + 改状态 + 产出事件)→ 广播 Event/State 给房内所有人(含观战)。客户端据事件渲染。**客户端发来的任何"结果"都不被信任,只接受其"意图"。**

### 3.1 房间生命周期
建房(房主,载入 GameConfig)→ 拿房间码/链接 → 受邀者输昵称占座(seat)→ 房主达人数点开始 → 进入对局 → 破产即转观战 → 到回合/时长上限或仅剩一人 → 结算排名 → 结果页/再来一局。

### 3.2 单个回合(普通流)
轮到玩家 P:启动**主时钟 25s** → P 掷骰(服务端 RNG)→ 引擎结算移动/落点(买地?盖房?付租?机会命运?监狱?)→ P 在窗口内做决策(买/不买/盖房/发起交易/抵押)→ 结束回合 → 轮转到下一位。主时钟超时 → 投递默认命令(不买地、自动付租、不主动交易)→ 结束回合。

### 3.3 交易(trade)—— PRD §4B ★
**前置:仅在 P 自己回合发起。** 子状态机如下:
1. P 发起报价(我方/对方地产 + 现金的组合)→ 引擎校验报价物归属合法 → **挂起主时钟**,进入 Trade 子状态,只接受"对方 Q 的响应"这一类命令,其它玩家命令拒绝/排队。
2. Q 限时 **20s**:接受 / 拒绝 / **还价(全局仅 1 次)**。
   - 接受 → 引擎**原子**校验双方资产与现金充足 → 一次性过户(地产 owner 变更 + 现金双向结算)→ 事件广播 → 退出子状态,**恢复主时钟**,回到 P 的回合。
   - 拒绝 / 20s 超时 → 交易作废 → 恢复主时钟,回到 P 回合。
   - 还价(仅 1 次)→ 计时切给 P,P 同样 **20s** 决定 接受/拒绝;P 接受则过户,P 拒绝或超时则作废。还价不可再被还价。
3. 任一方超时 = **自动拒绝**(PRD 口径)。掉线方由 AI 托管按"被动一律拒绝"处理。
> 关键:整个交易期间**主回合时钟暂停**,交易跑独立时钟(D-004),不会出现回合超时与交易超时同时触发。

### 3.4 拍卖(auction)—— PRD §4B ★
**触发:P 落到无主地产但选择不买/无力购买。**
1. 引擎对**全体在局玩家**开启 Auction 子状态 → 挂起主时钟 → 确定加价轮转顺序(自 P 的下家起)。
2. 轮到的玩家限时 **10s**:加价 或 放弃。
   - **出价原子性 [硬约束]**:引擎单写者串行校验 `出价 > 当前最高价 且 出价 ≤ 该玩家现金`;不满足则拒绝该出价(视为无效,可在 10s 内重出);满足则**原子**更新"当前最高价 + 最高出价人"。因命令串行,**两人同价/超额出价不可能发生**。
   - 放弃 / 10s 超时 = 本轮该玩家放弃;掉线/托管者直接放弃(PRD §4C)。
3. **连续一轮无人加价 → 成交**给当前最高价者(扣现金 + 过户)。**全员放弃 → 流拍**(地产维持无主)。退出子状态,恢复主时钟,回到 P 回合。

### 3.5 抵押(mortgage)—— PRD §4B
**仅自回合、无跨玩家冲突,最简单。** P 在自己回合对已有地产发起:引擎校验归属 → 抵押(给现金 ≈ 地产价 50%,标记被抵押→暂不可收租);赎回需付 抵押价 + 10% 利息。属普通回合内的同步状态变更,无需子状态机。

### 3.6 断线重连(reconnect)—— PRD 硬约束
权威状态常驻服务端内存(非客户端),故**玩家资产/进度不随掉线丢失**。流程:网关心跳判定掉线 → 标记该 seat 离线、在 ≤1 回合内启用 **AI 托管**接管,**不卡全场** → 玩家持 session token 在宽限期内重连 → 服务端推送当前**全量状态快照** → 客户端 rehydrate 渲染 → 玩家**收回控制权**。超过宽限期/到结算仍未回 → AI 托管持续至本局结束。

### 3.7 回合超时兜底
见 §2 定时器:超时不旁路改状态,而是合成"默认操作"命令入队,与真人命令同路。保证"超时行为"也走引擎裁决、可被快照与审计覆盖。

## 4. 数据模型(概念层,栈无关)

- **Room**:roomId、roomCode、status(lobby/playing/ended)、config(GameConfig 快照)、seats[]、spectators[]、turnState、createdAt。
- **Seat/Player**:seatId、displayName(匿名昵称,身份抽象②的当前实现)、cash、ownedPropertyIds[]、position(格子)、inJail/jailTurns、status(active/bankrupt/spectating)、connection(online/offline/ai-controlled)。
- **BoardTile**:index、type(property/start/jail/gotojail/freeparking/chance/fate/tax)、对 property 类型挂 PropertyState。
- **PropertyState**:tileId、groupId(彩色组)、ownerSeatId|null、houseLevel(0–4 + 旅馆)、mortgaged(bool)、baseRent/rentTable(取自 config)。
- **TurnState**:currentSeatId、phase(normal/trade/auction/mortgage)、mainTimerDeadline、subInteraction(TradeOffer | AuctionState | null)。
- **TradeOffer**:initiatorSeatId、counterpartySeatId、offerItems(give/get 的地产+现金)、step(awaitingResponse/countered)、subTimerDeadline、counterUsed(bool)。
- **AuctionState**:tileId、participants[]、bidOrder[]、currentHigh、currentHighSeatId|null、currentBidderSeatId、roundNoRaiseCount、subTimerDeadline。
- **Deck**:chance/fate 卡堆与弃牌堆。
- **GameConfig(§4A 参数化)**:mapTiles、groups、startingCash、rentCurve、housePriceRatio、passStartBonus、jailRules、turnTimerSec、tradeTimerSec、auctionTimerSec、hardTimeLimitMin、maxTurnsPerPlayer……**全部可配置**,建房载入。
- **ConnectionSession**:sessionToken、roomId、seatId、lastSeen(用于重连鉴权与宽限期判定)。

## 5. 对外接口/契约

### 5.1 实时消息协议(端无关,Web/H5 与小程序共享)
传输用 WebSocket,消息体 JSON、语言/端无关。两类:
- **Intent(客户端→服务端)**:`JoinRoom`、`StartGame`、`RollDice`、`BuyProperty`、`DeclineBuy`、`BuildHouse`、`Mortgage`/`Redeem`、`ProposeTrade`、`RespondTrade(accept|reject|counter)`、`PlaceBid`、`PassBid`、`EndTurn`、`Reconnect`……
- **Event/State(服务端→客户端,广播)**:`StateSnapshot`(全量,重连/进房用)、`DiceRolled`、`PlayerMoved`、`RentPaid`、`PropertyBought`、`TurnChanged`、`TradeProposed`/`TradeResolved`、`AuctionStarted`/`BidPlaced`/`AuctionResolved`、`PlayerBankrupt`、`GameEnded`、`PlayerOffline`/`PlayerReconnected`……
原则:Intent 只表达"想做什么",所有结果以服务端 Event 为准。

### 5.2 三个预留口子(§7.3,首版只留接口不实现)
- **① 可嵌入(iframe/SDK)**:客户端可独立加载;定义一个 host↔game 的嵌入接口(如 postMessage/JS-SDK 桩):宿主可传入启动参数、接收"对局结束"回调。首版给抽象与桩,不接真实宿主。
- **② 外部账号**:在网关身份层定义 `IdentityProvider` 抽象,首版实现 = 匿名昵称;将来替换为"社区登录态"无须改引擎(引擎只认 seatId)。
- **③ 合规层**:在网关连接/进房路径预留一个**拦截 seam**(实名/防沉迷 hook),首版 no-op 直通;转 A 面向公众时挂上即可。

### 5.3 配置契约
`GameConfig` 作为建房入参(或默认模板 + 覆盖项),是产品调参与架构之间的契约;§4A 默认值落为默认模板。

## 6. 备选方案与取舍

| 议题 | 选用 | 淘汰项 | 为什么 |
|------|------|--------|--------|
| 裁决位置 | 服务端权威 + 瘦客户端 | 客户端/P2P 裁决 | PRD 硬约束:骰子绝不在客户端;防作弊只能服务端裁决 |
| 房间并发模型 | 单房间单写者(有序命令队列/actor) | 共享存储 + 行锁/乐观并发 | 熟人低并发下,单写者最简且天然消除交易同价/拍卖超额竞态;省去分布式锁 |
| 时钟模型 | 主时钟与子交互时钟**互斥挂起** | 主时钟与交易时钟并行 | PRD ★ 要求隔离;并行会"双重超时打架" |
| 交易发起时机 | 仅自回合(随 PRD) | 任意时刻可发起 | 大幅降低状态同步复杂度(回合制里制造"实时孤岛"最小化) |
| 持久化 | 内存主存 + 快照 | 全量事件溯源(event sourcing) | 快照即可满足重连/崩溃恢复;事件日志留作二期审计/回放再加 |
| 超时处理 | 合成默认命令入队 | 定时器旁路直接改状态 | 让引擎始终是唯一事实源,超时行为同样可快照/可审计 |

## 7. 风险与未决

- **[风险] 交易/拍卖是复杂度高地**:建议开发顺序上**先搭回合主循环 + 这三个子状态机(尤其交易/拍卖的超时与掉线兜底)**作为最高风险件先打通,配套穷举超时/掉线/同价/超额用例(回流 05-acceptance 作长期回归)。
- **[风险] 单写者需要粘性路由**:房间与实例需 sticky;实例崩溃需快照恢复——快照节奏建议"每次权威变更后或每回合"落一次,具体在 03-tech-stack 选型时定存储与节奏。
- **[需确认 / 端无关] 微信小程序传输差异**:小程序有自己的 WebSocket API 与域名白名单要求。协议设计已做到端无关,但需各端写传输适配层;这点连同首版主攻 Web/H5、小程序后置,记入 03-tech-stack。
- **[待 03-tech-stack] 具体技术栈未定**:语言/运行时、WebSocket 框架、状态存储、部署形态尚未拍板;本设计的架构性决策(服务端权威、单写者、ws 协议、内存+快照、参数化规则集)已在 03 落为 ADR-lite(D-001~D-005),候选实现待选型确认,不在本文替团队假设语言。
- **[已确认 2026-05-30]** 单写者"按房间分片 + 粘性路由"在熟人低并发下作为起步取舍,**客户已点头**;将来转 A 面向公众放量时再评估扩展策略(关联 03-tech-stack D-001)。
