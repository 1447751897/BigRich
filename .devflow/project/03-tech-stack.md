# 技术选型

> 本文先固化 02-design 推导出的**架构性决策(ADR-lite)**;具体语言/框架/存储/部署的"产品级选型"待与开发工程师确认后填表(不在此替团队假设语言)。
> 标注:`accepted` = 已定;`proposed` = 架构倾向、待选型会确认。

## 1. 选型总表
| 领域 | 选用 | 候选 | 选择理由 |
|------|------|------|----------|
| 语言/运行时 | **TypeScript + Node.js (≥20)**(accepted 2026-05-30,开发工程师) | Go / Java / Python | 端无关核心可被 Web/H5 与小程序复用同一份纯 TS 引擎;长连接/内存态友好;全栈单语言降低协议漂移 |
| 实时传输 | **WebSocket(`ws`)**(accepted 2026-05-30) | SSE、长轮询 | 双向、低延迟、Web/H5 与小程序均支持;见 D-002 |
| 规则引擎形态 | **纯逻辑库 / 端无关核心**(accepted) | 与框架耦合的逻辑 | 端无关(§7.4)、可单测、可被多端复用;见 D-003 |
| 房间并发模型 | **单房间单写者(actor/有序队列)**(accepted) | 共享存储+锁、乐观并发 | 低并发下最简且天然消竞态;见 D-001 |
| 存储 | **内存主存 + JSON 快照**(accepted 2026-05-30) | 关系库为主、全事件溯源 | 满足重连/崩溃恢复即可;首版内存即权威态,快照经 `RoomRuntime.onSnapshot` seam 落本地 JSON;见 D-005 |
| 测试 | **vitest** | jest | 纯逻辑引擎单测;§4B 回归集 29 用例 |
| 部署/运行 | **单实例起步 + 按房间分片粘性路由**(accepted) | 多实例共享存储 | 单写者需 sticky;熟人低并发起步从简,放量转 A 再评估 |

## 2. 关键决策记录(ADR-lite)

### D-001 房间并发:单房间单写者(per-room single-writer)
- 状态:accepted
- 日期:2026-05-30
- 背景:PRD §4B 要求交易/拍卖的"出价原子性、防同价/超额"。需要一个能从根上消除竞态的并发模型,且匹配熟人低并发。
- 选项与结论:选"每房间一条有序命令队列、单实例串行处理";淘汰"共享存储 + 行锁/乐观并发"(对低并发是过度工程,引入分布式锁复杂度)。
- 影响:交易/拍卖原子性天然成立;需粘性路由(sticky)与崩溃后快照恢复;水平扩展靠按房间分片(关联 D-005、部署行)。
- 客户确认:2026-05-30,客户点头"按房间分片 + 粘性路由"作为起步取舍(放量转 A 再评估扩展)。

### D-002 实时传输:WebSocket
- 状态:**accepted**(2026-05-30,开发工程师确认)
- 日期:2026-05-30
- 背景:对局需服务端向房内多人低延迟广播状态,且客户端随时上行意图。
- 选项与结论:选 WebSocket(双向、低延迟),Node 端用 `ws` 库;淘汰 SSE/长轮询(上行弱/延迟高)。微信小程序有自有 WebSocket API 与域名白名单,需各端写传输适配层,协议层保持端无关(关联 D-003)。Intent/Event 信封为 JSON、端无关。
- 影响:网关层 `src/gateway/server.ts` 已落地骨架;各端传输适配;小程序上线前需处理白名单。

### D-003 规则引擎与渲染/传输解耦(端无关核心)
- 状态:accepted
- 日期:2026-05-30
- 背景:PRD §7.4 端无关——Web/H5 与小程序共享同一后端与协议。
- 选项与结论:规则引擎做成纯逻辑 `(状态,命令)→(新状态,事件)`,不依赖传输/渲染/系统时钟;淘汰"逻辑与框架耦合"。
- 影响:可单测、可多端复用;时钟以"超时→合成命令入队"方式外置(关联 D-004)。

### D-004 计时:主回合时钟与子交互时钟互斥挂起
- 状态:accepted
- 日期:2026-05-30
- 背景:PRD §4B ★ 要求交易计时与回合计时隔离不打架。
- 选项与结论:进入交易/拍卖子流程时挂起主时钟、跑独立子时钟,结束再恢复;淘汰"两时钟并行"(会双重超时)。超时不旁路改状态,而是合成默认命令入队。
- 影响:回合状态机含 trade/auction/mortgage 子状态;定时器服务为唯一权威时钟。

### D-005 持久化:内存主存 + JSON 快照
- 状态:**accepted**(2026-05-30,开发工程师确认)
- 日期:2026-05-30
- 背景:断线重连要求"资产/进度不丢";单写者实例崩溃需可恢复。
- 选项与结论:权威状态常驻内存,**每次权威变更后**经 `RoomRuntime.onSnapshot` 回调落一次快照;首版介质用**本地 JSON**(熟人低并发够用,转 A 放量再换持久层)。淘汰"全量事件溯源"作首版(过重,留二期做审计/回放)。
- 节奏:每条命令处理后触发 onSnapshot(变更驱动),崩溃恢复依赖最近一次快照新鲜度;`GameState` 全量可 `structuredClone`/JSON 序列化,重连推全量快照 rehydrate。
- 影响:`onSnapshot` seam 已在 RoomRuntime 落地;**落盘已实现**(`FileSnapshotStore`,本地 JSON,temp+rename 原子写),网关每次权威变更后保存、启动恢复未结束房间、旧 token 可重连。已知限制:崩溃恢复时长硬上限按完整 30 分钟重计(未持久化剩余时间)。

## 3. 约束与前提
- **硬约束(PRD)**:服务端权威(骰子/资金/结算/成交全部服务端裁决);端无关(Web/H5 与小程序共享后端与协议);三口子(嵌入/外部账号/合规层)只留接口不实现;实时兜底(房间同步、断线重连、回合超时、AI 托管)。
- **规模前提**:首版熟人量级、单房 ≤8 人、低并发;按此选"单写者 + 从简部署",非面向公众放量。
- **合规**:首版按"不公开运营"做,不实现实名/防沉迷,仅在网关留 no-op seam(③)。
