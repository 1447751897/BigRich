# 产品需求(PRD)

> 本文件起草于**需求澄清官**的《需求澄清纪要》(issue CUI-14),由**产品经理**细化为可交付 PRD。
> 标注约定:`[已确认]` = 客户已拍板;`[产品定稿]` = 产品经理基于纪要给出的可执行决策(默认值,可在 MVP 试玩中微调,客户如不认可可推翻);`[需架构对齐]` = 产品定义交互/口径,技术实现由架构师拍板;`[待客户确认]` = 仍需客户点头的少数项。
> 状态 `in_review`:产品细化已完成,待客户/小组对 §4–§7 的产品定稿项签字后升 `done`,再转 02-design(架构师)。

## 1. 背景与问题
做一款 **接入轻、实时可联机的"大富翁类"多人对战游戏**:免下载、免注册,点链接/扫码即开局,朋友间开房同玩。
首版面向熟人小范围、不公开商业运营;未来作为**内嵌小游戏接入客户自有的"轻社区"**,届时转向面向公众(版号/实名/防沉迷等合规届时再处理)。

**要解决的核心问题**:把"大富翁"这种单局偏长、易陪跑的桌游,改造成**朋友间随时开一把、人人玩到结束、约 20–30 分钟收束**的实时联机轻量局。

## 2. 目标与非目标
- **目标 [已确认]**:做出可玩的实时联机大富翁,2–8 人开房同局对战,单局可控收束、人人玩到结束不掉队。
- **产品目标(可度量)[产品定稿]**:
  - 一局完整闭环(开房→邀请→对战→结算→排名)可稳定跑通;
  - 8 人局也能在 **≤30 分钟**收束;
  - 任一玩家掉线/挂机**不卡全场**(自动兜底接管 ≤1 个回合内生效)。
- **首版非目标 [已确认]**(均待"转 A 面向公众"时再做):
  - 商业化 / 内购(皮肤、骰子、抽卡等)
  - 陌生人随机匹配 / 段位
  - 排行榜 / 赛季 / 养成
  - 实名认证 / 未成年人防沉迷

## 3. 目标用户与场景
- **首版用户 [已确认]**:熟人 / 朋友小圈子(内部、不公开)。
- **核心场景 [已确认]**:房主建房 → 分享房间码/链接 → 2–8 人实时同局对战 → 一局内分出排名。
- **演进场景 [已确认]**:将来嵌入客户的"轻社区"作为内嵌小游戏,面向社区公众用户。
- **典型使用画面 [产品定稿]**:
  1. **房主**点"创建房间" → 拿到房间码/分享链接 → 发到群里;
  2. **受邀者**点链接,输入一个昵称即进房(无需注册),房主满足人数后点"开始";
  3. 进入实时对局,轮流掷骰/买地/盖房/收租,可发起交易/拍卖,触发机会卡/进监狱;
  4. 有人破产即出局转**观战**;到回合/时长上限或仅剩一人时**结算排名**;
  5. 结果页展示名次与净资产,可"再来一局"。

## 4A. 玩法数值基线(产品定稿,供架构/开发起步;均为可调参数,MVP 试玩中调平衡)
> 这些是 §6 第 1 条"数值参数"的产品答复。给的是**起步默认值 + 设计意图**,实现上必须做成配置项(便于试玩调参),不要写死。

| 参数 | 起步默认值 | 设计意图 / 备注 [需架构对齐=★] |
|------|-----------|------------------------------|
| 地图格子数 | **28 格**(4 边 × 7 格) | 比经典 40 格小,跑圈更快→更快触发买地竞争与破产,利于 20–30 分钟收束 |
| 地产分组 | **6 组彩色地产**(每组 3–4 块)+ 4 个角(起点/监狱/免费停车/入狱)+ 机会/命运/税收若干 | 凑齐整组才能盖房,制造交易/拍卖动机 |
| 起始资金 | **1500 金币** | 约可买 2–3 块中价地产;偏紧→促成交易与抵押 |
| 过路费曲线 | 空地基础租金;**集齐整组×2**;盖房逐级跳涨(1/2/3/4 房 ≈ 基础租 ×5/×15/×40/×60),顶级"旅馆"再翻倍 | 越往后差距越大,加速淘汰、避免长尾陪跑 |
| 房价 | 与地产档位挂钩,**每栋房≈该地产价的 50–70%** | 控制盖房节奏 |
| 过起点奖励 | **+200 金币** | 经典口径,维持现金流 |
| 监狱 | 掷双数/付费/用卡出狱,最多停 3 回合 | 标准规则 |
| 单局**时长硬上限** | **30 分钟**(到点立即按净资产排名结算) | §6 第 2 条答复:这是 8 人局收束的主兜底 |
| 单局**回合上限** | 每位玩家 **20 个完整回合**(先到者触发结算) | 与时长上限取**先到先结算**,双保险 |
| 每回合操作**倒计时** | **25 秒**(掷骰/决策);超时执行默认操作(不买地、自动付租、不主动交易) | 防一人慢拖全场 |

## 4B. 交易 / 抵押 / 拍卖 —— 交互与超时口径(产品定稿 + 需架构对齐)
> §6 第 4 条答复。产品侧定义**交互流程与超时兜底**;并发原子性、状态同步、防作弊由架构师在 02-design 落实(标 ★)。

- **抵押(mortgage)[产品定稿]**:仅在**自己回合**操作。抵押已有地产换现金(抵押价 ≈ 地产价 50%),被抵押地产暂不可收租;赎回需付抵押价 + 10% 利息。无跨玩家实时冲突,优先级最简单。
- **交易(trade)[产品定稿 / ★]**:
  - 发起时机:**仅在发起方自己的回合**发起,避免与他人操作交叉(降低同步复杂度);
  - 报价内容:可组合"我方/对方的地产 + 现金";
  - 对方响应:限时 **20 秒**内 接受 / 拒绝 / **还价(仅 1 次)**;还价后发起方同样限时 20 秒决定;
  - 超时口径:任一方超时 = **自动拒绝**,交易作废,回到发起方回合继续;
  - ★ 发起交易期间,**发起方回合倒计时暂停**,交易有独立计时,避免双重超时打架(实现细节交架构)。
- **拍卖(auction)[产品定稿 / ★]**:
  - 触发:玩家落到无主地产但**选择不买**(或无力购买)时,该地产对**所有在局玩家**公开拍卖(经典规则,避免地产沉睡);
  - 形式:**轮流加价**,每轮每人限时 **10 秒**,可加价或放弃;**连续一轮无人加价则成交**给当前最高价者;全员放弃则流拍(地产维持无主);
  - ★ 出价的原子性(防止两人同价/超额出价超过现金)由服务端权威裁决。
- **共同口径**:以上所有交互**不得卡死全场**——任何一步超时都有明确的"自动作废/自动放弃"兜底;掉线玩家由 AI 托管按"不主动发起、被动一律拒绝交易 / 拍卖不加价"处理。

## 4C. AI 托管强度(产品定稿)
> §6 第 3 条答复。**首版简单规则 AI 足够**——它的职责是"防卡场",不是"当强敌"。

- 触发:玩家掉线、回合超时连续 N 次(默认 2 次)、或开局人数不齐需补位时,由 AI 接管。
- 行为(规则启发式,无需 AI 模型):
  - 掷骰/移动:正常进行;
  - 买地:现金 > 安全线(默认保留 ≥1 次预估过路费缓冲)时**买得起就买**,否则跳过;
  - 盖房:集齐整组且现金充裕时按从低到高补房;
  - 交易:**不主动发起**,收到报价一律拒绝;
  - 拍卖:不加价(直接放弃);
  - 出狱:有出狱卡/付得起就尽快出狱。
- 玩家回来(断线重连)后**收回控制权**;到结算仍未回则 AI 持续托管至本局结束。
- (二期可选)托管强度升级为更"会玩"的 AI,首版不做。

## 5. 成功指标 / 验收方向
- **完整一局闭环可跑通**:2–8 人完成 开房 → 邀请 → 实时对战(含买地/盖房/收租/机会卡/**交易/抵押/拍卖**)→ 破产淘汰或兜底封顶结算 → 出排名结果。
- **收束达标**:8 人局在 **30 分钟硬上限**前正常结算或到点强制结算,不存在打不完的局。
- **实时性可用**:掉线能**重连回原局**(资产/进度不丢,≤数秒恢复);掉线/超时由 **AI 托管或默认操作**兜住,接管在 ≤1 回合内生效,**不卡全场**。
- **防作弊**:掷骰、资金、地产结算、交易/拍卖成交全部**服务端裁决**(骰子绝不在客户端生成)。
- **交互不死锁**:交易/拍卖任一步超时都能自动作废/放弃并回到正常对局。

## 7. 关键产品决策明细
### 7.1 收束机制(为什么 30 分钟 + 20 回合双上限)
8 人 + 淘汰制最大风险是"打不完 / 早出局干等"。用**时长硬上限**保证一定收束、**回合上限**作为节奏兜底,二者先到先结算;配合**观战**让出局者不退场也不卡场。

### 7.2 去 IP 化(P1,硬约束)
命名、棋盘配色、卡牌文案、货币名一律避开"大富翁"/Monopoly 商标元素(如不用经典棕→深蓝配色与地名),为将来转 A 公开运营留干净底子。

### 7.3 架构预留三口子(P1,只留接口不实现)
① 可被外部页面嵌入(iframe/SDK);② 玩家身份可对接外部账号(将来用社区登录态替换匿名昵称);③ 可加挂实名/防沉迷拦截层。首版仅在身份/嵌入边界留好抽象,不实现具体逻辑。

### 7.4 端无关原则(支撑 Web/H5 与小程序同步)
核心规则逻辑(掷骰/结算/交易/拍卖)集中在**服务端权威**,客户端只做渲染与交互;Web/H5 与微信小程序共享同一套后端与协议,降低双端适配成本。(技术实现交架构)

### 7.5 美术风格 [待客户确认]
默认**简洁扁平卡通风**,移动端优先,首版用极简自绘/开源素材控制成本;不追求经典大富翁的拟物棋子/建筑动画手感(客户未把美术列为优先项)。客户如有偏好可在此推翻。

## 风险与提醒(已与客户对齐)
- ⚠️ **交易 + 拍卖纳入首版**会明显增加实时同步与防作弊复杂度(澄清阶段曾建议放二期,客户明确要求首版做)——工期与架构复杂度相应上调,§4B 已给产品口径,实现复杂度请架构师重点评估。
- ⚠️ **Web/H5 与微信小程序同步开发**需要"端无关"架构(核心规则逻辑与渲染解耦、共享后端),否则后期适配成本高。
- ⚠️ 将来**转 A(面向公众)**前,版号 / 实名 / 防沉迷是硬前置,是上线中国大陆公众的卡点;首版按 C 不做,但需在架构上为其预留(§7.3)。
