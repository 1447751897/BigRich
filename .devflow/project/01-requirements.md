# 产品需求

> 本文件由**需求澄清官**基于与客户确认的《需求澄清纪要》(issue CUI-14)填充。
> `[已确认]` = 客户已明确拍板;`[假设待确认]` = 澄清官的合理补全,待产品/架构核实。
> 状态 `in_review`:待产品经理接手细化并评审后升 `done`。

## 1. 背景与问题
做一款 **接入轻、实时可联机的"大富翁类"多人对战游戏**:免下载、免注册,点链接/扫码即开局,朋友间开房同玩。
首版面向熟人小范围、不公开商业运营;未来作为**内嵌小游戏接入客户自有的"轻社区"**,届时转向面向公众(版号/实名/防沉迷等合规届时再处理)。

## 2. 目标与非目标
- **目标 [已确认]**:做出可玩的实时联机大富翁,2–8 人开房同局对战,单局可控收束、人人玩到结束不掉队。
- **首版非目标 [已确认]**(均待"转 A 面向公众"时再做):
  - 商业化 / 内购(皮肤、骰子、抽卡等)
  - 陌生人随机匹配 / 段位
  - 排行榜 / 赛季 / 养成
  - 实名认证 / 未成年人防沉迷

## 3. 目标用户与场景
- **首版用户 [已确认]**:熟人 / 朋友小圈子(内部、不公开)。
- **核心场景 [已确认]**:房主建房 → 分享房间码/链接 → 2–8 人实时同局对战 → 一局内分出排名。
- **演进场景 [已确认]**:将来嵌入客户的"轻社区"作为内嵌小游戏,面向社区公众用户。

## 4. 核心需求(按优先级)
| 优先级 | 需求 | 说明 |
|--------|------|------|
| P0 | 接入轻 [已确认] | 免下载、免注册,点链接/扫码即玩 |
| P0 | 实时联机 + 好友开房 [已确认] | 实时同时在线;房主建房、分享房间码/链接进入 |
| P0 | 2–8 人对局 [已确认] | 单局支持 2 至 8 人 |
| P0 | 核心玩法 [已确认] | 掷骰移动、买地、收过路费、盖房/升级、机会/命运卡、监狱、破产判定 |
| P0 | 玩家交易 / 抵押 / 拍卖 [已确认] | 玩家间自由交易、地产抵押、流拍拍卖 —— **首版同步实现**(客户明确要求) |
| P0 | 胜负:淘汰制 + 兜底封顶 [已确认] | 破产即出局;到回合/时长上限时强制结束,按净资产(现金+地产估值)排名 |

## 5. 成功指标
- **完整一局闭环可跑通**:2–8 人完成 开房 → 邀请 → 实时对战(含买地/盖房/收租/机会卡/**交易/抵押/拍卖**)→ 破产淘汰或兜底封顶结算 → 出排名结果。
- **实时性可用**:掉线能**重连回原局**(资产/进度不丢);掉线/超时由 **AI 托管或默认操作**兜住,**不卡全场**。
- **防作弊**:掷骰、资金、地产结算全部**服务端裁决**(骰子绝不在客户端生成)。
- **单局时长可控**:[假设待确认] 即便 8 人局也建议能在 ~20–30 分钟内收束(由兜底封顶上限数值决定)。

## 6. 待确认
> 不阻塞起步,交产品/架构细化:
- [ ] 数值参数:地图格子数、起始资金、回合/时长上限、过路费与房价曲线。
- [ ] 单局目标时长上限的具体取值(决定 8 人局收束速度)。
- [ ] AI 托管强度:首版"简单规则 AI"是否够用。
- [ ] **实时交易 / 拍卖**的交互与超时处理细节(回合制里的"实时孤岛",需架构重点设计同步与原子性/防作弊)。
- [ ] 美术风格:客户未把"美术轻"列为优先项;暂假设简洁卡通棋盘风,待产品确认。
- [ ] 房间容量上限与并发规模预估(首版熟人量级小)。

## 风险与提醒(已与客户对齐)
- ⚠️ **交易 + 拍卖纳入首版**会明显增加实时同步与防作弊复杂度(澄清阶段曾建议放二期,客户明确要求首版做)——工期与架构复杂度相应上调,请架构师重点关注。
- ⚠️ **Web/H5 与微信小程序同步开发**需要"端无关"架构(核心规则逻辑与渲染解耦、共享后端),否则后期适配成本高。
- ⚠️ 将来**转 A(面向公众)**前,版号 / 实名 / 防沉迷是硬前置,是上线中国大陆公众的卡点;首版按 C 不做,但需在架构上为其预留。
